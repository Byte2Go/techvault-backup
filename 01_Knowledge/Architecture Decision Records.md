# Architecture Decision Records (ADRs)
An **Architecture Decision Record (ADR)** is a lightweight document that captures a significant <mark style="background: #ABF7F7A6;">architectural decision</mark>, including its <mark style="background: #D2B3FFA6;">context, reasoning, tradeoffs, and consequences</mark>. It serves as the "<mark style="background: #ADCCFFA6;">living history" of a system's evolution</mark>.

---
## 🧭 Why ADRs Exist

Architectural decisions are easily forgotten over time leading to recurring questions:

- _Why did we <mark style="background: #ADCCFFA6;">choose Kafka over RabbitMQ</mark> two years ago?
- _Why did we <mark style="background: #ADCCFFA6;">build a microservices architecture instead of modular monolith</mark>?
- _Why did we accept <mark style="background: #ADCCFFA6;">eventual consistency for the checkout service</mark>?
- _Why did we <mark style="background: #ADCCFFA6;">migrate from Oracle to PostgreSQL</mark>?

<mark style="background: #FF5582A6;">Without an ADR, the technical context is lost</mark>, and <mark style="background: #FFB8EBA6;">teams waste time re-debating old choices</mark>. An ADR permanently preserves this organizational memory.

---
## 📋 A Typical ADR

A standard ADR follows a highly structured, scannable template:

| **Section**                     | **Purpose**                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Title**                       | ADR 004: Two-Tier Circuit Breaking for Public Ingress                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **Status**                      | Approved                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Context & Problem Statement** | Our public-facing Front-End API service has experienced occasional memory saturation, causing container threads to freeze. When this happens, public requests experience long hang times, and AWS API Gateway continues to flood the failing containers with traffic, preventing self-recovery and causing cascading timeouts across the system. We need a resilient mechanism to protect end-users from waiting and to give degraded containers room to recover.                                                                                                                                                                                                                                                                                                       |
| **Considered Options**          | <mark style="background: #FFF3A3A6;">1. **Option 1:** </mark>Application-level circuit breaking using Resilience4j inside the Java application. <mark style="background: #FFF3A3A6;">2. **Option 2:**</mark> Distributed serverless state machines via AWS Step Functions and DynamoDB. <mark style="background: #FFF3A3A6;">3. **Option 3:**</mark> Two-Tier Edge-and-Mesh resilience using AWS API Gateway timeouts coupled with Istio Ingress Outlier Detection..                                                                                                                                                                                                                                                                                                    |
| **Decision Outcome**            | Chosen Option: **Option 3 (Two-Tier Edge-and-Mesh resilience)**. We will offload infrastructure-level resilience entirely to the platform layer, keeping our application code clean and avoiding complex serverless infrastructure orchestration for basic edge protection.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Implementation Blueprint**    | 1. **Edge Ceiling (AWS API Gateway):** Configure a hard 2000ms integration timeout on the Front-End HTTP integration to ensure clients never experience an unresponsive connection. 2. **Cluster Ingress Circuit Breaker (Istio):** Apply a `DestinationRule` targeting the front-end service inside Kubernetes to isolate and temporarily eject containers failing with consecutive 5xx errors.                                                                                                                                                                                                                                                                                                                                                                        |
| **Consequences**                | ### Positive (Pros) * **Zero Code Overhead:** No java-level library maintenance or annotation clutter inside the microservices. * **Proactive Recovery:** Outlier detection actively silences traffic to degrading pods, allowing Kubernetes liveness probes to cycle them without dropping requests. * **Low Latency:** Eliminates the network hop and execution cost of using an intermediate Step Function for edge routing. ### Negative / Risks (Cons) * **Dual-Plane Configuration:** Infrastructure changes now span both CloudFormation/Terraform (AWS API Gateway) and Kubernetes GitOps manifests (Istio CRDs). * **Generic Fallbacks:** Edge timeouts will result in standard HTTP 504 gateway errors rather than context-aware application fallback states. |

### 3 Golden Rules for Using ADRs in Production
1. **Keep them in Git:** Store your ADRs in a /docs/adr folder inside your project's primary Git repository. <mark style="background: #FFF3A3A6;">Treat them like code</mark>—<mark style="background: #D2B3FFA6;">they should go through the standard Pull Request (PR) review process.</mark>
2. **Immutable History:** <mark style="background: #FF5582A6;">Never edit an old ADR to change a decision.</mark> <mark style="background: #FFB86CA6;">If your architecture changes a year from now, create a new ADR (e.g., ADR 012) with a status of Supersedes ADR 004</mark>. This preserves the historical timeline of _why_ things evolved.
3. **Focus on the "Why", Not the "How":** Code repositories tell you _how_ something works. <mark style="background: #BBFABBA6;">ADRs tell you _why_ it was built that way instead of using an alternative approach.</mark>
