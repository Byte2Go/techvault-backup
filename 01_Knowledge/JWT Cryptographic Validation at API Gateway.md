### 1. The Async Secret Sharing (The Setup)
The Auth Server (Identity Provider like Okta, Keycloak, or Auth0) holds a **Private Key** which it uses to sign tokens. It exposes its corresponding **Public Key** via a standard, public URL endpoint called the **JWKS (JSON Web Key Set)** endpoint.
- During boot-up (or at regular cached intervals), the **API Gateway** calls this JWKS endpoint _once_ and caches the Public Key locally in its own memory.

### 2. The Offline Verification (The Request)
When a user sends an API call with the JWT in the header, the Gateway pulls the token and validates it right there:
- **Step A (Signature Check):** The Gateway mathematically verifies the JWT's signature using its locally cached **Public Key**. If the signature is intact, it proves the token was generated by your Auth Server and has not been tampered with.
- **Step B (Claims Check):** The Gateway reads the plain-text JSON payload inside the JWT to check the expiration timestamp (`exp`) and the target audience (`aud`).

If both steps pass, the Gateway forwards the request to your microservices. **The Auth Server was never contacted during this API call.**

**Architectural Rule:** The API Gateway validates JWT Access Tokens **statelessly and offline**. It downlinks the Auth Server's **Public Key** via a JWKS endpoint, caches it at the edge, and uses it to verify token signatures instantly without introducing a network bottleneck.
